What Just Happened?

  1. The client-side Angular application asked the server-side ASP.NET application to create a viewing session for a specific document (in this case, {{filename}}):

  2. The ASP.NET application POSTed to PAS (PrizmDoc Application Services) to create a new viewing session (in this sample, the application server is an ASP.NET application, but your actual application server could be anything, like Java, Node.js, etc.):

  3. The ASP.NET application received a viewingSessionId of "{{viewingSessionId}}":

  4. The ASP.NET application sent the viewingSessionId to the client-side Angular application so that it could create the HTML viewer control:

  5. The ASP.NET application uploaded the source document to PAS, associating that document with the viewing session:

  6. PAS handed this work off to the powerful PrizmDoc Server backend, which immediately began converting the document content to SVG, one page at a time:

  7. Meanwhile, as soon as it had finished loading in the browser, the viewer began repeatedly asking PAS for the first page of document content:

  8. As soon as the first page of SVG content was ready, PAS returned it to the viewer, allowing you to see the first page of the document:

  9. As you navigate through the document, the viewer will continue to request page content as needed.

How Does the Viewer Reach PAS (PrizmDoc Application Services)?

To get document content, the viewer makes requests to PAS through your web application (or web server). That's why the ASP.NET application defines a proxy route to PAS at pas-proxy/ (see Startup.cs, where we use SharpReverseProxy in the Configure method to setup a pas-proxy/ base route which proxies requests to the actual PAS host). If you look at the network traffic in dev tools, you'll see the requests made by the viewer for resources rooted at this path.
